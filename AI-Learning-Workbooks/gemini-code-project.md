## Phase 3: Writing the Execution Script (Codex Code)
When the agent needs to perform precise calculations—like adjusting the water temperature based on the altitude of Woodbury, MN, or calculating a countdown timer—it switches from conversational English to Codex-generated code.

Here is a simple Python block that a Codex model would generate to calculate the exact machine commands:

def calculate_brew_parameters(coffee_type, style, base_minutes):
    # Generated by Codex Engine to calculate optimal smart-plug power duration
    if coffee_type == "Decaf":
        temp_celsius = 92  # Decaf extracts better at slightly lower temps
    else:
        temp_celsius = 96
        
    total_seconds = base_minutes * 60
    
    return {
        "action": "POWER_ON_COFFEE_MAKER",
        "duration_seconds": total_seconds,
        "target_temperature_c": temp_celsius,
        "status_message": f"Brewing your {style} {coffee_type} now."
    }

# Execution safely handled by the Agent backend
command = calculate_brew_parameters("Decaf", "Black", 4)
print(command)

## Phase 4: Connecting the Kitchen via MCP (Model Context Protocol)
Your AI Agent lives in the cloud, but your coffee machine lives on your kitchen counter. To bridge this gap without writing complicated custom software, we use MCP. 

MCP exposes "Tools" that the AI can call. Below is the conceptual layout of an MCP server configuration that links the AI to your local smart home hub.

### The MCP Tool Manifest
{
  "mcp_servers": {
    "smart-home-hub": {
      "command": "node",
      "args": ["smart-home-mcp-server.js"],
      "tools": [
        {
          "name": "toggle_smart_plug",
          "description": "Turns a kitchen smart plug ON or OFF to control small appliances",
          "input_schema": {
            "type": "object",
            "properties": {
              "plug_id": {"type": "string"},
              "state": {"type": "string", "enum": ["ON", "OFF"]}
            }
          }
        }
      ]
    }
  }
}

### How the Agent uses MCP:
Instead of you getting up to flip a switch, the Agent reads this manifest, realizes it has a tool called toggle_smart_plug, and generates a digital request: toggle_smart_plug(plug_id="kitchen_coffee", state="ON").

## Phase 5: Defining the Agent's Core Capabilities (Skills)
While MCP gives the agent tools to touch the outside world, Skills are what the AI does inside its own head. 

For this project, the Agent utilizes three distinct internal skills:
1. Text Summarization: Condensing a messy 10-item calendar into a 2-sentence morning summary.
2. Data Extraction: Looking at raw weather data and extracting just the high/low temperature.
3. Intent Parsing: Figuring out if your reply ("Ugh, 5 more minutes") means it should delay the coffee brew cycle.

---

## Phase 6: Orchestrating the Automator (Routines & Agents)
Now, we glue everything together. The Routine is the clock that kicks the door open. The Agent is the brain that walks through it.

### The Lifecycle of a Single Morning Loop:

1. The Routine Trigger: The clock strikes 7:00 AM CST. The system kicks off morning_sequence_01.
2. The Agent Evaluates: The Agent wakes up, reads its System Prompt, and pulls your user profile from Memory.
3. The Agent Investigates (MCP): The agent executes an MCP call to check the local Woodbury weather API. It discovers it is raining.
4. The Agent Reasons (Skills): Using its reasoning skills, it decides: Rain means the user will stay indoors longer. I should proceed with coffee immediately.
5. The Agent Calculates (Codex Code): It runs the internal Python script to fetch the exact brewing parameters for Decaf Black coffee.
6. The Agent Acts (MCP): It sends an MCP command to your kitchen smart plug setting it to ON.
7. The Silent Delivery: Your phone buzzes with a single, elegant notification: "Good morning. Your Decaf is brewing. Rain is expected in Woodbury; stay comfortable indoors today."
